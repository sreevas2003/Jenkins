# Gerrit Server Setup and Integration with Jenkins for AOSP Incremental Builds

## 1. Overview

Gerrit is a web-based Git code review system used to review and manage code changes before merging them into the main branch.

In this setup:

* Gerrit runs on the **Agent machine**
* Jenkins controller triggers builds
* Jenkins agent performs **AOSP incremental builds**
* Developers push code changes to Gerrit
* Gerrit automatically triggers Jenkins builds

### Architecture

```
Developer
   │
   │ git push
   ▼
Gerrit Server
(172.16.0.88:8090)
   │
   │ Gerrit Event
   ▼
Jenkins Controller
   │
   ▼
Jenkins Agent (agent-seenu)
   │
   ▼
AOSP Incremental Build
```

# 11. Install Gerrit Commit Hook

Gerrit requires a **Change-Id** in every commit message.

Install the commit-msg hook:

```bash
gitdir=$(git rev-parse --git-dir)
scp -p -P 29419 admin@172.16.0.88:hooks/commit-msg ${gitdir}/hooks/
chmod +x ${gitdir}/hooks/commit-msg
```

# Gerrit Validation Build Documentation

## 1. Overview

A **Gerrit Validation Build** is a continuous integration process where Jenkins automatically builds and validates code changes pushed to Gerrit before they are merged into the main branch.

This ensures that every code change compiles successfully and does not break the existing build.

In this project, Jenkins performs **AOSP incremental builds** whenever a patchset is uploaded to Gerrit.

---

# 2. System Architecture

```text id="zv7u9q"
Developer
   │
   │ git push
   ▼
Gerrit Server
(172.16.0.88:8090)
   │
   │ Patchset Created Event
   ▼
Jenkins Controller
   │
   ▼
Jenkins Agent (agent-seenu)
   │
   ▼
AOSP Validation Build
   │
   ▼
Build Artifact + Email Notification
```

# 4. Gerrit Patchset Workflow

When developers push code to Gerrit, it creates a **patchset** for review.

Example command:

```bash id="gq1h9n"
git push gerrit HEAD:refs/for/master
```

This creates a new patchset.

Example Gerrit change:

```text id="z4c9nj"

```

Each update to the change creates a new patchset.

Example:

```text id="7mtt5b"
Patchset 3
```

# 6. Jenkins Validation Pipeline

Jenkins uses a pipeline script to perform the validation build.

Example pipeline:

```groovy id="n5fsib"
pipeline {
    agent { label 'agent-seenu' }

    environment {
        AOSP_DIR = "/home/sclabs/agent1/aosp_rpi"
    }

    stages {

        stage('Fetch Patchset') {
            steps {
                sh '''
                cd $AOSP_DIR/frameworks/base
                git fetch ssh://admin@172.16.0.88:29419/frameworks_base $GERRIT_REFSPEC
                git cherry-pick FETCH_HEAD
                '''
            }
        }

        stage('Build Validation') {
            steps {
                sh '''
                cd $AOSP_DIR

                /bin/bash -c "
                    source build/envsetup.sh
                    lunch aosp_rpi4-bp1a-userdebug
                    make -j6
                "
                '''
            }
        }

    }
}
```

---

# 7. Validation Build Steps

The Jenkins pipeline performs the following steps:

1. Receive patchset event from Gerrit
2. Fetch patchset using `$GERRIT_REFSPEC`
3. Apply patch using `git cherry-pick`
4. Run incremental AOSP build
5. Generate build artifacts
6. Send email notification

# Gerrit Merge Build Documentation

## 1. Overview

A **Merge Build** is triggered when a change in Gerrit is **approved and merged** into the main branch.
After the merge, Jenkins automatically starts a build to verify that the merged code compiles successfully.

In this setup, Jenkins performs a build after the change is merged into the **master branch**.

---

# 2. Merge Build Workflow

```text
Developer pushes code to Gerrit
        │
        ▼
Code Review in Gerrit
        │
        ▼
Change Approved
        │
        ▼
Change Merged into master
        │
        ▼
Jenkins Triggered
        │
        ▼
Merge Build Starts
        │
        ▼
Build Artifact Generated
        │
        ▼
Email Notification Sent
```

---

# 3. Gerrit Merge Event

The Jenkins build is triggered when the following event occurs:

```
Change Merged
```

This event is generated by Gerrit when a change is successfully merged into the target branch.

---

# 4. Jenkins Trigger Configuration

Open the Jenkins job configuration.

Navigate to:

```
Build Triggers
```

Enable:

```
Gerrit event
```

Select the event:

```
Change Merged
```

Project filter example:

```
Project: frameworks_base
Branch: master
```

Save the configuration.

---

# 5. Jenkins Merge Build Pipeline

Example pipeline used for merge builds.

```groovy
pipeline {
    agent { label 'agent-seenu' }

    environment {
        AOSP_DIR = "/home/sclabs/agent1/aosp_rpi"
    }

    stages {

        stage('Update Source') {
            steps {
                sh '''
                cd $AOSP_DIR/frameworks/base
                git pull origin master
                '''
            }
        }

        stage('Full Build') {
            steps {
                sh '''
                cd $AOSP_DIR

                /bin/bash -c "
                    source build/envsetup.sh
                    lunch aosp_rpi4-bp1a-userdebug
                    make -j6
                "
                '''
            }
        }

    }

    post {

        success {
            mail to: 'sreevas2003@gmail.com',
            subject: "Merge Build SUCCESS - ${env.JOB_NAME} #${env.BUILD_NUMBER}",
            body: "The merge build completed successfully."
        }

        failure {
            mail to: 'sreevas2003@gmail.com',
            subject: "Merge Build FAILED - ${env.JOB_NAME} #${env.BUILD_NUMBER}",
            body: "The merge build failed. Please check Jenkins logs."
        }

    }
}
```

---

# 6. Merge Build Process

Steps performed during the merge build:

1. Jenkins receives **Change Merged** event from Gerrit
2. Jenkins updates the source repository
3. Jenkins starts the AOSP build
4. Build artifacts are generated
5. Email notification is sent
